    1.1 项目背景与目标

    了解 my-blog 项目的起源、设计目标和适用场景

    目录

    项目起源

    my-blog 项目经历了从 JavaScript/TypeScript 到 Rust，最终选择 Zig 的演进过程。

    前身项目

    早期的博客系统基于 JavaScript 技术栈： - 前端: Preact + CSS Modules - 后端: Node.js 静态文件服务 - 构建: Webpack

    这种架构存在以下问题： 1. 运行时依赖 - 需要 Node.js 环境 2. 构建速度慢 - Webpack 打包耗时 3. 类型安全弱 - JavaScript 动态类型易出错

    为什么选择 Zig？

    在评估了多种方案后，最终选择 Zig 作为重写语言，主要基于以下考虑：

    因素 Zig 优势
    性能 编译型语言，与 C 性能相当
    简洁 语言特性少，易于掌握
    无隐藏控制流 代码可读性强
    内存安全 手动管理，无 GC 开销
    编译速度 比 Rust 快得多
    C 互操作 可直接调用 C 库

    设计目标

    主要目标

    1. 高性能生成
    2. 编译为原生二进制，无运行时依赖

    3. 批量处理 Markdown 文件，快速生成 HTML

    4. 简洁易用

    5. 单一命令完成构建和生成
    6. 清晰的目录结构

    7. 直观的 Markdown 语法支持

    8. 现代化设计

    9. 响应式布局，适配移动端
    10. 暗色/亮色主题自动切换

    11. 专业的排版和 typography

    12. 可扩展性

    13. 模块化设计
    14. 自定义模板系统
    15. 易于添加新功能（RSS、搜索等）

    非目标

    以下功能不在当前设计范围内：

    • ❌ 动态内容（评论、用户系统）
    • ❌ 多语言支持（i18n）
    • ❌ 实时预览服务器
    • ❌ 插件系统

    适用场景

    ✅ 适合的场景

    场景 说明
    技术博客 支持代码高亮、数学公式
    个人网站 简洁专业的设计
    项目文档 Markdown 编写，版本控制友好
    学习笔记 标签分类，易于整理

    ❌ 不适合的场景

    场景 原因
    电商网站 缺少动态功能
    论坛/社区 无用户系统
    新闻门户 缺少内容管理系统

    与其他静态博客生成器对比

    Hexo (Node.js)

    特性 my-blog Hexo
    运行时 无 Node.js
    安装 编译即可 npm install
    生成速度 快 中等
    主题系统 简单 丰富
    插件生态 无 丰富

    Hugo (Go)

    特性 my-blog Hugo
    语言 Zig Go
    二进制大小 ~500KB ~20MB
    编译速度 极快 快
    功能完整性 基础 完整
    学习曲线 低 中等

    Jekyll (Ruby)

    特性 my-blog Jekyll
    运行时 无 Ruby
    依赖 无 多个 gem
    构建速度 快 慢
    GitHub Pages 需手动部署 原生支持

    各阶段特点

    阶段 技术栈 特点
    2019-2021 JavaScript 动态类型，Webpack 构建
    2021-2023 Rust 类型安全，编译慢
    2023-2026 Zig 性能与简洁的平衡

    小结

    my-blog 是一个用 Zig 编写的现代化静态博客生成器，旨在提供：

    • ✅ 高性能 - 原生二进制，无运行时依赖
    • ✅ 简洁性 - 单一命令，清晰结构
    • ✅ 现代化 - 响应式设计，主题切换
    • ✅ 可扩展 - 模块化设计，易于定制
